# ADR-009: Mount Everest Product Loop (M7)

- **Status:** Accepted
- **Date:** 2026-07-21
- **Pillars:** [Z] [D] [A] ([V] architecture preserved; no Proven Core expansion without a new ADR)

## Context

M6 closed the **production-ready bar** on Latitude/QEMU (HA, harden, soak, external audit/spec review). That is not the same as a **shippable single-host operator product**:

> Ship the EFI → boot via iDRAC virtual media on a **real PowerEdge R640** → **network-reachable** Web UI → **register a Linux ISO / create VM / install guest**.

Lived gates closed M0–M6 primarily on Latitude + QEMU. `CLAUDE.md` M0 said “boots on R640”; that iron claim remains open. M5 “full mgmt plane” shipped as an in-process SPA/REST shape without a host NIC HTTP listener. Arbitrary ISO install was never an M0–M6 numbered gate.

Cluster features (vMotion-like live migrate, DRS-like placement, hot-add) are commercially desirable but must not block the single-host loop. Ghost page-transfer proofs (M6.3 / ADR-004) are substrate, not product vMotion.

## Decision

1. **Open milestone M7 — Mount Everest** in `CLAUDE.md` as the shippable **single-host** product loop. Living tracker: [`docs/hda.md`](../hda.md) · plan: [`docs/m7_plan.md`](../m7_plan.md).
2. **M7 definition of done** (all required):
   - **E1 Ship EFI** — versioned `.efi` + checksums + size gate + USB/iDRAC vMedia runbook  
   - **E2 R640 boot** — marker on **real** PowerEdge R640 (Latitude/QEMU insufficient to close M7)  
   - **E3 Network UI** — browser on operator LAN reaches HTTPS (or HTTP) SPA + REST  
   - **E4/E5 Datastore + ISO install** — image library; ISO register; CD-ROM or documented extract-boot; virtio-blk install target; create-VM path in UI  
3. **M7 cannot close on Latitude alone.** QEMU/Latitude may wire and partially prove software gates; **`RAYNU-V-R640-BOOT-OK` (or equiv.) is mandatory for M7 closed.**
4. **Cluster / elasticity = M8 (sketch only):** vMotion-like live migrate UI/ops, DRS-like placement, hot-add CPU/RAM/disk. Out of M7 critical path.
5. **Proven Core boundary:** HTTP/TLS, datastore, ISO/CD-ROM, and Web UI remain **outside** Proven Core unless a future ADR explicitly expands the boundary (default **no**). Size still governed by ADR-003.
6. **Pre-iron build order** (R640 expected ~1 month after this ADR):  
   **Ship kit → TLS/HTTP → datastore/ISO** (software ready before iron week).

## Consequences

- Numbered M6 stays closed; further Everest work is M7 (then M8), not silent M6.10.
- HDA months-to-Everest tracks M7 residual; update `docs/hda.md` + `./tools/sync-hda-site.sh` on every Everest-path commit.
- ADR-005 Tier‑1 vMedia + serial is on the M7 critical path; live Redfish health may follow first R640 boot; Tier‑2 never blocks M7.
- ADR-004 migrate page-transfer proofs do **not** imply product vMotion; product live-migrate is M8.
- ADR-007 inventory import remains separate from M7 ISO *install* path.
- Binary growth from HTTP + ISO + UI must stay inside ADR-003 (lazy/zstd; hard 20 MB).
- **M7.1 lab note:** plaintext HTTP is accepted to close the network mgmt gate; TLS is
  deferred under the size budget. UEFI NIC listen may stub (`UnsupportedOnFirmware`)
  while the in-binary codec + host TCP proof green the host/CI gate — firmware bind
  follows Tcp4/SNP (or equivalent) without reopening the codec GAP.
