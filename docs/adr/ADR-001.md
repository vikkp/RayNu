# ADR-001: Why Verus + Kani

- **Status:** Accepted
- **Date:** 2026-06-01
- **Pillars:** [V]

## Context

RayNu-V requires formal assurance for the Proven Core (VMX, VMCS, EPT, allocator, etc.). A single tool does not cover both deep functional correctness and soundness of `unsafe` blocks equally well. We need a dual-tool strategy that never blocks shipping when one tool lags.

## Decision

Use **Verus** (Microsoft Research) as the primary SMT-based verifier for functional specs and proofs on Proven Core modules. Use **Kani** (Amazon) as the secondary tool for bounded model checking of `unsafe` blocks. Keep runtime assertions and fuzzing as the permanent fallback.

Fallback chain: Verus → Kani → runtime assertions + fuzzing. Never blocked.

## Rationale

- Verus aligns with ghost state, pre/post conditions, and the L2→L3 maturity path (ADR-006).
- Kani targets the small `unsafe` surface with `// KANI-TARGET` tags.
- Dual coverage reduces R09 (wrong specs) and R08 (proof effort) by allowing L1/L2 ship while proofs catch up.

## Consequences

- CI must eventually run `cargo verus --verify` and `cargo kani` (soft-fail acceptable until toolchain is installed).
- Every Proven Core module follows the four-file convention (`*_spec.rs`, `*_proof.rs`, `*_test.rs`).
- Toolchain pins live in `rust-toolchain.toml` and `verus-version.toml` (ADR-008).
