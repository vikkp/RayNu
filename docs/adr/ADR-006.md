# ADR-006: Verification Maturity Model

- **Status:** Accepted
- **Date:** 2026-06-01
- **Pillars:** [V]

## Context

Not every Proven Core module reaches full proof at the same time. We need a shared vocabulary so "verified" is not binary, and so tooling gaps never freeze product delivery.

## Decision

| Level | Name | Meaning |
|-------|------|---------|
| L0 | Documented Invariants | Invariants in code/doc comments only |
| L1 | Runtime-Enforced | `assert!` / `debug_assert!` at entry/exit; Kani for unsafe |
| L2 | Spec-Written | Verus `*_spec.rs` with ghost state, pre/post |
| L3 | Proof-Complete | `cargo verus --verify` passes for all inputs |

Rules:

- Architecture is always designed for L3 even when tooling is not ready.
- L1 with extensive fuzzing is far safer than unverified code.
- Runtime assertions are **kept** at L3 (defense-in-depth).
- Ship at L1/L2 if Verus blocks L3; never blocked (fallback chain ADR-001).

## Consequences

- PRs must not regress a module's maturity level.
- New Proven Core functionality adds L0 docs immediately; L1 asserts when behavior lands; L2 specs when stable enough.
- Milestone gates report a verification checkpoint (which modules at which level).
