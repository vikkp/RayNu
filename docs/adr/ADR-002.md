# ADR-002: Proven Core Boundary

- **Status:** Accepted
- **Date:** 2026-06-01
- **Pillars:** [V]

## Context

Formally verifying an entire Type-1 hypervisor is intractable. We must draw a hard boundary around security-critical modules so verification effort stays focused and the LOC budget remains honest.

## Decision

Only the following modules are **inside** the Proven Core (~12,000 LOC planned, **15,000 LOC hard limit** including scaffolding):

- VMX lifecycle, VMCS management
- EPT engine, physical frame allocator
- vCPU state management, interrupt injection
- Hypercall interface, MSR/CPUID/CR firewalls
- Audit log integrity, IPI confinement

Everything else (devices, vSwitch, mgmt plane, migrate, scheduler algorithms, iDRAC, Linux boot protocol, audit *report* generation) stays **outside** and relies on Rust's type system + testing only.

## Scope Creep Rules

1. No module enters the Proven Core without a formal ADR.
2. Default answer is always "outside."
3. Promotion requires LOC budget update + timeline impact assessment.
4. Demotion requires an ADR explaining why verification was abandoned.

## Consequences

- Inside modules use the four-file convention; outside modules use standard `module.rs` + `module_test.rs`.
- Changes that enlarge the core without an ADR are rejected in review.
- ~3,000 LOC buffer is for proof scaffolding growth or at most one small promotion.
