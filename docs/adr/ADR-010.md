# ADR-010: Living Verification Paper as Progressive Public Artifact

- **Status:** Accepted
- **Date:** 2026-07-21
- **Pillars:** [V] [A]

## Context

RayNu-V’s long-term differentiator is formal verification of the Proven Core
(ADR-002, ADR-004, ADR-006). The Verification Maturity Model already requires
progressive, honest advancement from L0 → L1 → L2 → L3.

Traditional papers are written only after results are complete. This leaves
progress invisible and risks retrospective narrative construction. We already
emit the raw material (serial logs, QEMU/R640 traces, Verus/Kani output,
audit-ring events, gate checklists). An external site (RaynuV.com) sits outside
the single-binary constraint and the Proven Core, making it a safe place for a
living document.

## Decision

1. Maintain a **living verification paper** structured and styled as a formal
   academic paper from day one. In-repo source of truth lives at
   `docs/paper/RayNu-V-Verification-Paper.md`; the public page on RaynuV.com
   renders the same content.
2. Sections are populated **only** with evidence from real execution logs,
   proof outputs, audit events, and milestone gates.
3. Maturity claims (L0/L1/L2/L3) on the page must never exceed the actual
   maturity of the corresponding Proven Core modules.
4. The document is versioned (`v0.1-skeleton`, `v0.2-M0`, … `v1.0-preprint`).
5. When Proven Core reaches sufficient L3 coverage + external spec review,
   a frozen snapshot becomes the conference / arXiv submission.
6. Non-Proven-Core components (management plane, VMware migration engine,
   device emulation, etc.) appear only in system-overview / appendix sections,
   never in formal verification claims (ADR-002, ADR-007).

The living page is an **audit artifact**, not marketing copy.

## Consequences

- Continuous, honest external visibility of verification progress.
- Stronger credibility with reviewers and partners; natural source for the
  M6 verification whitepaper.
- Temptation to fill sections prematurely must be resisted: evidence blocks
  are mandatory; maturity on the page ≤ maturity in code.
- Page updates become part of the relevant milestone gate checklist.
- Version history + corresponding hypervisor commit / proof hash must be
  preserved so earlier claims remain auditable.
- Zero impact on the 15 MB / 20 MB single-binary budget (ADR-003).
- Proof maintenance (ADR-008) still applies to the underlying Verus proofs;
  the paper simply reports current state.
