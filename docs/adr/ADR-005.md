# ADR-005: iDRAC Integration Tiers

- **Status:** Accepted
- **Date:** 2026-06-01
- **Pillars:** [D]

## Context

Dell PowerEdge R640/R650/R660 is the first-class hardware target. Some iDRAC/Redfish capabilities are standard and self-sufficient; others need Dell partnership or reverse-engineering and must not block milestones.

## Decision

Split iDRAC work into two tiers:

### Tier 1 — Low Risk (Self-Sufficient)

- COM1 serial via iDRAC virtual console
- Redfish: thermal, fan, PSU status
- SMBIOS: DIMM topology, NUMA layout
- ACPI MADT / SRAT / SLIT
- Intel X710 NIC awareness; NVMe direct passthrough

### Tier 2 — High Risk (Partnership / RE)

- PERC H740P RAID health via Dell OEM Redfish
- DIMM SPD beyond standard SMBIOS
- Predictive failure alerts (OEM schemas)
- Auto-throttle VMs on hardware degradation

**Rule:** Never block a milestone on Tier 2. Tier 1 is always sufficient to ship.

## Consequences

- `idrac/` module implements Tier 1 first; Tier 2 is optional and feature-gated.
- Hardware CI may use QEMU for non-iDRAC paths; R640 iron validates Tier 1 serial/Redfish.
- Risk R07 is accepted: Tier 2 slip does not slip M0–M4 (or M7 — see ADR-009).
- **M5.6:** closed on Latitude via documented mock Redfish + SMBIOS/ACPI topology
  (`assets/idrac/`, marker `RAYNU-V-M5-IDRAC-OK`). Live BMC HTTP and Tier‑2 OEM stay GAPs.
- **M7 (ADR-009):** Tier‑1 **deploy path** is on the critical path — USB or iDRAC virtual media
  boot of `r640-hypervisor.efi` + COM1/iDRAC serial to close `RAYNU-V-R640-BOOT-OK`.
  Live Redfish health readback may follow first R640 boot; Tier‑2 never blocks M7.
